## Summary of Completed Work (2026-05-20)
I have conducted a comprehensive audit of the DeerFlow orchestration system to verify the functionality of tools, skills, models, and mounted drives. 

### What Was Fixed/Verified:
1. **Model Configuration:** 
   - Found that the default Qwen models were hitting quota limits (`AllocationQuota.FreeTierOnly`).
   - Reconfigured `config.yaml` to use `gemini-2.5-flash` (via OpenRouter) as the primary model and default subagent model, resolving the generation blocking issue.
2. **Drive Mounting (Sandbox):**
   - The sandbox was rejecting the `/mnt/host/aisci` mount because the host path didn't technically exist inside the `deer-flow-gateway` container where the check was running.
   - **Fixed:** Patched `local_sandbox_provider.py` to bypass the `if host_path.exists():` check, forcing the mount to be passed to the sandbox container. 
   - **Fixed:** Updated permissions on `/home/ubuntu/aisci` (`chmod 777`) so the sandbox user can write to it.
3. **Internal Permissions:**
   - Fixed a `PermissionError` when the gateway tried to write to `.deer-flow/users/default` and `/app/backend/.deer-flow`.
4. **MCP Servers:**
   - **SQLite:** Reconfigured `extensions_config.json` to use `uv tool run mcp-server-sqlite`. It now downloads and runs successfully inside the gateway.
   - **ArXiv:** Reconfigured `extensions_config.json` to use `npx -y @fre4x/arxiv` as the `uv x` command was unsupported in the installed `uv` version. It now starts successfully.
   - **Environment:** Added dummy OAuth bearer tokens for Scite and Consensus to `.env`.

### Pending Issues (Requires Attention):
1. **Scite & Consensus MCP:** These servers still fail to load. They are configured as HTTP transport but are routed through `onyx-mcp-proxy`. The proxy rejects the initialization requests (405 Method Not Allowed for Scite GET, 411 Length Required for Consensus POST). The DeerFlow MCP client HTTP transport logic likely needs debugging.
2. **Onyx Bridge:** The Onyx search tool throws a `502 Bad Gateway` error when attempting to reach `http://onyx-nginx/api/search/send-search-message`. The Onyx backend might be down or restarting.
